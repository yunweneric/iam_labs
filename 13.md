# Lab 13: SELinux Booleans and Ports

## Objective

The objective of this lab is to learn how SELinux controls which network ports a service may use. You will view the ports Apache is permitted to bind to, change Apache to listen on a non-default port, observe the resulting failure, then authorize the new port with `semanage` and restore normal operation.

---

## Environment

- Operating System: CentOS 7, AlmaLinux 8, or AlmaLinux 9
- Tools Used:
  - Apache (`httpd`)
  - `semanage`
  - `systemctl`
  - `/var/log/messages`

---

## Prerequisites

Complete [Lab 12](12.md) first, or ensure Apache (`httpd`) is already installed and running on this VM.

---

## Step 1: View Authorized Apache Ports

List the TCP ports SELinux allows the Apache web server daemon to use.

```bash
sudo semanage port -l | grep http
```

### Expected Result

Port 80 (and typically 443) appear in the list with the type `http_port_t`.

**Screenshot 1:** Authorized http ports.

---

## Step 2: Change Apache to Listen on Port 82

Open the Apache configuration file.

```bash
sudo nano /etc/httpd/conf/httpd.conf
```

Find the line:

```text
Listen 80
```

Change it to:

```text
Listen 82
```

Save and exit.

Restart Apache:

```bash
sudo systemctl restart httpd
```

### Expected Result

Apache fails to restart because SELinux does not yet allow binding to port 82.

**Screenshot 2:** Apache restart failure.

---

## Step 3: View the Error Message

Check the system log for SELinux-related errors.

```bash
sudo tail -20 /var/log/messages
```

### Expected Result

The log contains a message indicating that Apache was denied permission to bind to port 82 due to SELinux policy.

**Screenshot 3:** SELinux denial in /var/log/messages.

---

## Step 4: Authorize Port 82

Add port 82 to the list of ports SELinux associates with HTTP services.

```bash
sudo semanage port -a -t http_port_t -p tcp 82
```

Verify the port was added:

```bash
sudo semanage port -l | grep http
```

Restart Apache:

```bash
sudo systemctl restart httpd
```

### Verification

```bash
sudo systemctl status httpd
```

### Expected Result

Apache starts successfully on port 82.

**Screenshot 4:** Apache running on port 82.

---

## Step 5: Remove the Custom Port

Delete the port entry you added.

```bash
sudo semanage port -d -t http_port_t -p tcp 82
```

Verify:

```bash
sudo semanage port -l | grep http
```

### Expected Result

Port 82 no longer appears in the `http_port_t` port list.

**Screenshot 5:** Port 82 removed.

---

## Step 6: Restore the Default Configuration

Edit the Apache configuration again.

```bash
sudo nano /etc/httpd/conf/httpd.conf
```

Change:

```text
Listen 82
```

back to:

```text
Listen 80
```

Save and exit.

Restart Apache:

```bash
sudo systemctl restart httpd
```

### Verification

```bash
sudo systemctl status httpd
```

### Expected Result

Apache returns to listening on port 80 and operates normally.

**Screenshot 6:** Apache restored to port 80.

---

## Results

SELinux port labeling was examined with `semanage port -l`. Changing Apache to port 82 without updating SELinux policy caused a bind failure. Adding port 82 to `http_port_t` allowed Apache to start. Removing the custom port and reverting the configuration returned the system to its original state.

---

## Conclusion

This lab showed that SELinux enforces not only file contexts but also **port access** for network services. Administrators must use `semanage port` to authorize non-standard ports before a service can bind to them. Understanding SELinux booleans and port contexts is critical when deploying web applications on custom ports in production environments.
