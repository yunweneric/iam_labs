# Lab 2: Hands-on Lab for Disabling the sudo Timer

## Objective

The objective of this lab is to understand how the **sudo authentication timeout mechanism** works and how it can be configured. By default, after entering a password once, a user can execute additional sudo commands for a short period without re-entering the password. This lab demonstrates how to disable that timeout globally and then configure it for a specific user only. 

---

## Environment

- Operating System: AlmaLinux 9
- Tool Used:
  - `sudo`
  - `visudo`
  - `sudoers` configuration file

---

# Step 1: Verify Normal sudo Behavior

Run the following commands from your normal user account:

```bash
sudo fdisk -l

sudo systemctl status sshd

sudo iptables -L
```

### Expected Result

You should only be prompted for your password once. Subsequent sudo commands execute without requiring another password because the sudo timer is active. 

📸 **Screenshot 1:** Execution of the three commands showing normal sudo behavior.

---

# Step 2: Reset the sudo Timer Manually

Run:

```bash
sudo fdisk -l

sudo -k

sudo fdisk -l
```

### Explanation

The command:

```bash
sudo -k
```

invalidates the current sudo authentication timestamp.

### Expected Result

When executing the second `sudo fdisk -l`, you will be prompted to enter your password again. 

📸 **Screenshot 2:** Password prompt after using `sudo -k`.

---

# Step 3: Open the sudoers File

Edit the sudoers configuration:

```bash
sudo visudo
```

Add the following line under the Defaults section:

```text
Defaults timestamp_timeout=0
```

Save and exit. 

📸 **Screenshot 3:** Modified sudoers file.

---

# Step 4: Test the New Configuration

Run:

```bash
sudo fdisk -l

sudo systemctl status sshd

sudo iptables -L
```

### Expected Result

You should now be required to enter your password every time you execute a sudo command because the timeout has been disabled globally. 

📸 **Screenshot 4:** Multiple password prompts.

---

# Step 5: Restrict the Timeout Setting to Lionel Only

Open the sudoers file again:

```bash
sudo visudo
```

Replace:

```text
Defaults timestamp_timeout=0
```

with:

```text
Defaults:lionel timestamp_timeout=0
```

Save and exit. 

📸 **Screenshot 5:** User-specific timeout configuration.

---

# Step 6: Compare User Behavior

### Test with Your Own Account

```bash
sudo fdisk -l

sudo systemctl status sshd

sudo iptables -L
```

Expected:

- Password requested once.

### Test with Lionel

```bash
su - lionel

sudo fdisk -l

sudo systemctl status sshd

sudo iptables -L
```

Expected:

- Lionel must enter a password for every sudo command.

📸 **Screenshot 6:** Comparison between normal user and Lionel.

---

# Step 7: View Assigned sudo Privileges

Run:

```bash
sudo -l
```

### Expected Result

The command displays all sudo permissions assigned to the current user. 

📸 **Screenshot 7:** Output of `sudo -l`.

---

# Results

The sudo authentication timeout was successfully modified:

- Initially, sudo cached credentials for a period of time.
- Using `sudo -k` cleared the cache manually.
- Setting `timestamp_timeout=0` required password entry for every sudo command.
- Restricting the setting to Lionel affected only that user.

---

# Conclusion

This lab demonstrated how sudo manages authentication caching and how administrators can control it using the `timestamp_timeout` parameter in the sudoers file. The exercise showed both global and user-specific configurations, improving understanding of privilege management and security hardening in Linux systems.
