# Lab 15: Blocking Invalid IPv4 Packets

## Objective

The objective of this lab is to improve firewall efficiency by dropping invalid IPv4 packets in the **mangle** table's `PREROUTING` chain before they reach the `filter` table. You will use Nmap NULL and Windows scans to generate suspicious traffic, observe packet counters, and compare behavior before and after adding the mangle rules.

---

## Environment

- Operating System: Ubuntu 22.04
- Tools Used:
  - iptables
  - nmap
  - iptables-save

---

## Prerequisites

Complete [Lab 14](14.md) on the same Ubuntu VM. The existing iptables rules from Lab 14 must remain in place.

---

## Important Note

Before starting, ensure you have a VM snapshot from Lab 14. This lab adds rules to the mangle table but does not replace your existing filter rules.

---

## Step 1: View Current Rules and Reset Counters

Display verbose rules for the filter and mangle tables, then zero the packet counters.

```bash
sudo iptables -L -v
```

```bash
sudo iptables -t mangle -L -v
```

```bash
sudo iptables -Z
```

```bash
sudo iptables -t mangle -Z
```

### Expected Result

The Lab 14 filter rules are present. Mangle table counters are reset to zero.

**Screenshot 1:** Initial verbose rule output.

From your host machine or another VM, run the following scans against your Ubuntu VM. Replace `ip_address_of_your_VM` with the actual IP.

```bash
sudo nmap -sN ip_address_of_your_VM
```

```bash
sudo nmap -sW ip_address_of_your_VM
```

### Expected Result

The scans complete. Invalid or unusual packets are processed by the existing filter rules.

**Screenshot 2:** Nmap scan output.

## Step 3: Observe Blocked Packet Counters

Repeat the verbose listing commands.

```bash
sudo iptables -L -v
```

```bash
sudo iptables -t mangle -L -v
```

### Expected Result

The packet counter on the final `DROP` rule in the `INPUT` chain of the filter table shows a large increase, indicating that invalid scan packets were blocked late in processing.

**Screenshot 3:** Increased DROP counter in filter table.

---

## Step 4: Add Mangle PREROUTING Rules

Add rules to drop invalid packets earlier in the Netfilter pipeline.

```bash
sudo iptables -t mangle -A PREROUTING -m conntrack --ctstate INVALID -j DROP
```

```bash
sudo iptables -t mangle -A PREROUTING -p tcp ! --syn -m conntrack --ctstate NEW -j DROP
```

### Explanation

| Rule | Purpose |
|------|---------|
| `ctstate INVALID` | Drops packets that do not belong to a valid connection |
| `tcp ! --syn` with `ctstate NEW` | Drops new TCP connections that do not begin with a SYN flag |

These rules stop malformed scan traffic before it reaches the filter table.

### Verification

```bash
sudo iptables -t mangle -L -v
```

**Screenshot 4:** New mangle PREROUTING rules.

## Step 5: Save the Configuration

Save the updated ruleset and copy it to the persistent rules directory.

## Results

Nmap NULL and Windows scans generated invalid packets that were initially blocked by the filter table's default DROP policy. After adding mangle PREROUTING rules, the same scans were blocked earlier in the Netfilter pipeline. Packet counters confirmed that each scan type triggered a different mangle rule.

---

## Conclusion

This lab demonstrated how the **mangle** table's `PREROUTING` chain can improve firewall performance by discarding invalid packets before they consume resources in the filter table. Dropping `INVALID` connections and TCP packets that are `NEW` but not `SYN` is a standard hardening technique that mitigates stealth port scans and malformed connection attempts.
