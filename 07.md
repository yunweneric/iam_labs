# Lab 7: Hands-on Lab for Configuring pam_tally2

## Objective

The objective of this lab is to learn how to use the **pam_tally2** module to protect Linux systems against brute-force password attacks. The module tracks failed login attempts and automatically locks user accounts after a specified number of unsuccessful login attempts. This helps prevent attackers from repeatedly guessing passwords.

---

## Environment

* Operating System: CentOS 7
* Tools Used:

  * PAM (Pluggable Authentication Modules)
  * pam_tally2
  * login authentication service

---

## Background

Brute-force attacks attempt to gain unauthorized access by repeatedly trying different passwords. The `pam_tally2` module mitigates this threat by:

* Counting failed login attempts
* Locking accounts after a defined number of failures
* Allowing administrators to reset lockouts when necessary

---

## Step 1: Configure pam_tally2 for Console Logins

Open the login PAM configuration file:

```bash
sudo nano /etc/pam.d/login
```

Locate the line containing:

```text
pam_securetty.so
```

Immediately below it, add:

```text
auth required pam_tally2.so deny=4 even_deny_root unlock_time=1200
```

Save the file and exit.

### Explanation

* `deny=4` : Lock account after four failed attempts.
* `even_deny_root` : Apply the policy to the root account.
* `unlock_time=1200` : Automatically unlock after 1200 seconds (20 minutes).

**Screenshot 1:** Modified `/etc/pam.d/login` file.

---

## Step 2: Configure pam_tally2 for System Authentication

Open:

```bash
sudo nano /etc/pam.d/password-auth
```

Add the following line above the first `auth required` entry:

```text
auth required pam_tally2.so deny=4 even_deny_root unlock_time=1200
```

Save and exit.

Repeat the same process for:

```bash
sudo nano /etc/pam.d/system-auth
```

Add:

```text
auth required pam_tally2.so deny=4 even_deny_root unlock_time=1200
```

Save and exit.

**Screenshot 2:** Modified PAM authentication files.

## Step 3: Trigger an Account Lockout

Log out of the current session.

Attempt to log in to the **samson** account using an incorrect password multiple times.

### Expected Result

After several failed attempts, the account becomes locked.

**Screenshot 3:** Account lockout message.

## Step 4: View Failed Login Counters

Log back into an administrator account.

Display the login failure counters:

```bash
sudo pam_tally2
```

### Expected Result

The output shows the number of failed login attempts recorded for each user.

**Screenshot 4:** Output of `pam_tally2`.

## Step 5: Reset a Locked Account

Simulate a help-desk account unlock procedure.

Reset Samson's failed login counter:

```bash
sudo pam_tally2 --user=samson --reset
```

Verify the reset:

```bash
sudo pam_tally2
```

### Expected Result

The failed login count for Samson returns to zero.

**Screenshot 5:** Resetting the account lockout.

---

## Step 6: Modify the Lockout Threshold

Open:

```bash
sudo nano /etc/pam.d/login
```

Locate:

```text
auth required pam_tally2.so deny=4 even_deny_root unlock_time=1200
```

Change it to:

```text
auth required pam_tally2.so deny=100 even_deny_root unlock_time=1200
```

Save and exit.

### Explanation

A threshold of 100 failed attempts is more realistic for production systems and reduces the likelihood of accidental account lockouts.

**Screenshot 6:** Updated lockout threshold.

The pam_tally2 module was successfully configured to monitor failed login attempts and automatically lock user accounts after repeated authentication failures. The account lockout was tested and reset successfully using administrative commands.

This lab demonstrated how PAM can be used to defend Linux systems against brute-force attacks. By implementing account lockout policies, administrators can significantly reduce the risk of password guessing attacks while maintaining the ability to unlock accounts when necessary.
