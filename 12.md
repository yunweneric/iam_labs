# Lab 12: SELinux Type Enforcement

## Objective

The objective of this lab is to install the Apache web server and SELinux administration tools, then observe how SELinux **type enforcement** affects file access. By deliberately assigning an incorrect SELinux context to a web content file, you will trigger an access denial and then restore the correct context using `restorecon`.

---

## Environment

- Operating System: CentOS 7, AlmaLinux 8, or AlmaLinux 9
- Tools Used:
  - Apache (`httpd`)
  - `setroubleshoot`
  - `setools`
  - `policycoreutils`
  - `chcon`
  - `restorecon`
  - `ls -Z`

---

## Step 1: Install Apache and SELinux Tools

On **CentOS 7**:

```bash
sudo yum install httpd setroubleshoot setools policycoreutils policycoreutils-python -y
```

On **AlmaLinux 8 or 9**:

```bash
sudo dnf install httpd setroubleshoot setools policycoreutils policycoreutils-python-utils -y
```

### Expected Result

Apache and the SELinux troubleshooting utilities install successfully.

**Screenshot 1:** Package installation.

---

## Step 2: Activate setroubleshoot

Restart the audit daemon so SELinux troubleshooting messages are captured.

```bash
sudo service auditd restart
```

### Expected Result

The `auditd` service restarts without errors.

**Screenshot 2:** auditd restart.

---

## Step 3: Enable Apache and Open Port 80

Start Apache and configure the firewall to allow HTTP traffic.

```bash
sudo systemctl enable --now httpd
```

```bash
sudo firewall-cmd --permanent --add-service=http
```

```bash
sudo firewall-cmd --reload
```

### Verification

```bash
sudo systemctl status httpd
```

### Expected Result

Apache is active and port 80 is open in the firewall.

**Screenshot 3:** Apache service status.

---

## Step 4: Create the Test Web Page

Create an `index.html` file in the Apache document root.

```bash
sudo nano /var/www/html/index.html
```

Add the following content:

```html
<html>
<head>
<title>SELinux Test Page</title>
</head>
<body>
This is a test of SELinux.
</body>
</html>
```

Save and exit.

### Expected Result

The test page is created in `/var/www/html/`.

**Screenshot 4:** index.html contents.

---

## Step 5: View the SELinux Context

Display the SELinux security context of the web file.

```bash
ls -Z /var/www/html/index.html
```

### Expected Result

The file context is `httpd_sys_content_t` (or equivalent), which is the correct type for Apache web content.

**Screenshot 5:** SELinux context of index.html.

---

## Step 6: Verify Normal Web Access

From your host machine's web browser, navigate to the IP address of the virtual machine.

Example:

```text
http://192.168.1.100
```

### Expected Result

The SELinux test page displays successfully in the browser.

**Screenshot 6:** Web page loads correctly.

---

## Step 7: Induce an SELinux Violation

Change the file type to an incorrect SELinux context.

```bash
sudo chcon -t tmp_t /var/www/html/index.html
```

Verify the change:

```bash
ls -Z /var/www/html/index.html
```

### Expected Result

The file type changes from `httpd_sys_content_t` to `tmp_t`.

**Screenshot 7:** Modified SELinux context.

---

## Step 8: Observe the Access Denial

Reload the page in your host machine's web browser.

### Expected Result

The browser displays a **Forbidden** message because Apache is not permitted to read files labeled `tmp_t` as web content.

**Screenshot 8:** Forbidden error in browser.

---

## Step 9: Restore the Correct Context

Use `restorecon` to reset the file to its default SELinux context.

```bash
sudo restorecon /var/www/html/index.html
```

Verify:

```bash
ls -Z /var/www/html/index.html
```

### Expected Result

The file context returns to `httpd_sys_content_t`.

**Screenshot 9:** Restored SELinux context.

---

## Step 10: Confirm Web Access Is Restored

Reload the page in your web browser.

### Expected Result

The test page displays successfully again.

**Screenshot 10:** Web page loads after restorecon.

---

## Results

Apache was installed and configured with the appropriate SELinux tools. The web page worked when the file had the correct SELinux type. Changing the type to `tmp_t` caused SELinux to block Apache from serving the file. Running `restorecon` returned the file to its default context and restored access.

---

## Conclusion

This lab demonstrated SELinux **type enforcement**, which restricts processes to accessing only files with compatible security contexts. Even when standard Unix permissions would allow access, SELinux can deny it if the file type does not match the policy. Tools such as `chcon`, `ls -Z`, and `restorecon` are essential for diagnosing and correcting SELinux context problems on web servers and other services.
