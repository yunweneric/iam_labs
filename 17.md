# Lab 17: Squid ACL Configuration

## Objective

The objective of this lab is to install and configure a **Squid** proxy server with multiple Access Control List (ACL) types, then test the rules from client systems. You will restrict access by source IP, destination domain, and port, and verify that Squid evaluates `http_access` rules in order from top to bottom.

---

## Environment

| VM | Role | Suggested OS |
|----|------|--------------|
| Proxy server | Squid proxy | Ubuntu Server 22.04 |
| Client 1 | Internal workstation | Ubuntu 22.04 |
| Client 2 | Internal workstation | AlmaLinux 9 |
| Client 3 | Internal workstation | CentOS 7 |

- Tools Used:
  - Squid (`squid`)
  - `squid -k parse`
  - `curl`
  - UFW or `firewalld`

---

## Network Layout

```text
[Ubuntu Client] ----\
[AlmaLinux Client] ---+---- [Ubuntu Server: Squid :3128] ---- Internet
[CentOS Client] -----/
```

All client VMs should be on the same subnet as the Squid server (for example, `192.168.1.0/24`).

## Step 3: Define ACLs by Source Network

Edit the Squid configuration.

```bash
sudo nano /etc/squid/squid.conf
```

Add the following ACL definitions near the top of the file (adjust the subnet to match your lab network):

```text
acl localnet src 192.168.1.0/24
acl localhost src 127.0.0.1/32
acl SSL_ports port 443
acl Safe_ports port 80 443 8080 8443
acl CONNECT method CONNECT
```

### Expected Result

Source-network and port ACLs are defined.

**Screenshot 3:** Source and port ACL definitions.

---

## Step 4: Define ACLs by Destination Domain

Add domain-based ACLs to the same file:

```text
acl blocked_sites dstdomain .facebook.com .twitter.com .tiktok.com
acl allowed_dev_sites dstdomain .github.com .stackoverflow.com
```

### Explanation

| ACL | Type | Purpose |
|-----|------|---------|
| `blocked_sites` | `dstdomain` | Matches social media domains |
| `allowed_dev_sites` | `dstdomain` | Matches developer resource sites |

**Screenshot 4:** Domain ACL definitions.

---

## Step 5: Define a Time-Based ACL

Add a time restriction ACL:

```text
acl work_hours time MTWHF 08:00-17:00
```

This ACL is active Monday through Friday between 08:00 and 17:00.

**Screenshot 5:** Time-based ACL.

---

## Step 6: Configure http_access Rules

Add the following `http_access` rules in order. Place them **above** any existing `http_access deny all` line:

```text
http_access deny !Safe_ports
http_access deny CONNECT !SSL_ports
http_access deny blocked_sites
http_access allow allowed_dev_sites localnet work_hours
http_access allow localnet
http_access allow localhost
http_access deny all
```

### Explanation

Squid evaluates rules sequentially and stops at the first match:

1. Deny traffic to unsafe ports.
2. Deny CONNECT requests to non-SSL ports.
3. Block social media sites for all users.
4. Allow developer sites for the local network during work hours.
5. Allow general local network access.
6. Allow localhost.
7. Deny everything else.

**Screenshot 6:** http_access rule order.

## Step 9: Test from the Ubuntu Client

Configure the client to use the proxy (replace the IP with your Squid server address).

```bash
export http_proxy=http://192.168.1.10:3128
export https_proxy=http://192.168.1.10:3128
```

Test access to an allowed site:

```bash

```

Test access to a blocked site:

```bash

```

### Expected Result

GitHub returns HTTP headers successfully. Facebook is denied by Squid.

## Results

Squid was configured with ACLs based on source network, destination domain, port, and time of day. `http_access` rules were applied in the correct order to enforce a default-deny policy with selective allowances. Client systems on Ubuntu, AlmaLinux, and CentOS successfully used the proxy, with blocked domains denied as expected.

---

## Conclusion

This lab demonstrated how Squid **Access Control Lists** provide application-layer access control for HTTP and HTTPS traffic. Unlike network firewalls that filter by IP and port, Squid can enforce policies based on domains, URLs, time, and authentication. Because rule order matters, administrators must place specific deny rules before broad allow rules and always end with `http_access deny all`.
